* Automatic Parsing
** Requirements

I found an awesome browser extension called [[https://chromewebstore.google.com/detail/furiganator/ijfcmmkkmogaclaafpgcjdfcfibmhcfa][Furiganator]] which can insert
Furigana (Ruby text) in the webpage with the following features:

- Very fast and reliable.
- Furigana in Hiragana, Katakana or Romaji.
- Option to show Furigana only on mouse over.
- Option to make the furigana text unselectable.
- Option to use color-tags to easily identify the kanji level or grade.
- Multiple color themes.

In its Licenses, Certificates and Acknowledgements it describes the technologies
used:

- Icons8 - Icons provided by https://icons8.com
- Kanji API and its dependencies - This Extension uses parts of [[https://github.com/onlyskin/kanjiapi.dev][Kanji API]] and by
  extension, EDICT and KANJIDICT from [[https://www.edrdg.org/][Electronic Dictionary Research and Development Group]].
- Kuromoji.JS and its dependencies - This Extension uses [[https://github.com/takuyaa/kuromoji.js][Kuromoji.JS]] as
  Morphological Analyzer.
- Kuroshiro and its dependencies - This Extension uses [[https://github.com/hexenq/kuroshiro][Kuroshiro]] for its core
  functionality, and by extension the [[https://github.com/hexenq/kuroshiro-analyzer-kuromoji][Kuroshiro-analyzer-kuromoji]]
- Webpack and its dependencies - This Extensions uses [[https://webpack.js.org/][Webpack]]

Now I want to implement the same features in our minimal-mistakes-plus plugin
with the same technologies. Specifically,

- It should be very fast and reliable.
- Use the the option ~furigana: true/false~ in the front matter to turn the
  whole furigana parsing process on (~true~) or off (~false~ or unset).

- It can add Furigana in Hiragana, Katakana or Romaji automatically to normal
  texts without the furigana pattern using the same technologies as Furiganator
  depending on the option ~furigana_mode~, which

  - applies to the whole document when specified in the front matter:

    #+begin_src text
    furigana_mode: hiragana/katakana/romaji
    #+end_src

  - applies to only the paragraphs marked by the attribute syntax:

    #+begin_src text
    {: furigana_mode="hiragana/katakana/romaji"}
    #+end_src

  The attribute syntax takes precedence if both applies to a paragraph.

  If the option is unset, default to ~hiragana~.

- Use the option ~furigana_selectable: true/false~ in the front matter to make
  the furigana text selectable (~true~) or unselectable (~false~ or unset).

- Use the option ~furigana_mouse_over: true/false~ to only show Furigana on
  mouse over (~true~) or to show it all the time (~false~ or unset).

- Option to use color-tags to easily identify the kanji level or grade (Do not
  implement).
- Multiple color themes (Do not implement).

*** Integration with Existing Furigana Parsing

For the existing furigana parsing process dealing with the pattern:

#+begin_src text
（X｜Y｜Z：X1｜Y1｜Z1）
#+end_src

its processing logic should be kept intact, with these additional changes:

- Now it belongs to the whole furigana parsing process which can be turned on or
  off using the option ~furigana: true/false~ as mentioned above.
- If it's turned off, convert all furigana patterns to normal texts without the
  pattern in the output, that is, ~（X｜Y｜Z：X1｜Y1｜Z1）~ becomes ~XYZ~.
- The pattern takes precedence over the automatic parsing, that is, a pattern
  should not be sent for automatic processing.

** Implementation
{: .ordered}

To achieve a seamless, lightning-fast, and precise automatic Furigana injection
using the exact same technologies as Furiganator, we need to architect a
*Ruby-to-Node.js bridge*.

Spawning a new Node.js process for every single paragraph or file would be
extremely slow because Kuromoji's dictionaries take hundreds of milliseconds to
load. Instead, the solution uses a *persistent Node.js worker* that boots up
once per Jekyll build, loads the dictionary, and sits idly waiting for arrays of
text jobs over =stdin=.

Here is the complete implementation with the required comments and explanations.

*** Install Node.js Dependencies

Navigate to your plugin's directory and initialize the Node dependencies.
Kuroshiro and its analyzer are strictly required for the Node worker.

#+begin_src sh
cd lib/minimal_mistakes_plus
npm init -y
npm install kuroshiro kuroshiro-analyzer-kuromoji
#+end_src

*** The Node.js Worker (=kuroshiro_worker.js=)

Create a new file =lib/minimal_mistakes_plus/kuroshiro_worker.js=. This is the
high-performance daemon that parses Japanese morphology via Kuromoji and
converts it to HTML Ruby text.

*** The Updated Ruby File (=furigana.rb=)

Overwrite your existing =lib/minimal_mistakes_plus/furigana.rb=.

This code acts as the director. It handles global/local settings, strips out
manual syntax when toggled off, acts as an isolation barrier for
=<code>=/=<pre>= tags so Kuroshiro doesn't parse code blocks, and securely
streams strings back and forth with the Node worker.

*** GitHub Actions

You should commit the following files:

#+begin_example
kuroshiro_worker.js
package.json
package-lock.json
#+end_example

You must *not* commit the =node_modules= folder. It is already included in your
=.gitignore= file, which is the correct standard practice.

However, because you are not committing =node_modules=, there is a hidden trap
for GitHub Actions: when your workflow pulls your plugin as a Ruby Gem to build
the site, the Node.js dependencies will be missing, causing the Jekyll build to
crash when it tries to spawn the worker.

To solve this flawlessly without bloating your Git repository or requiring users
to modify their GitHub Actions YAML files, you can add a tiny "auto-install"
hook to your Ruby script. GitHub Actions (~ubuntu-latest~) has Node.js
pre-installed, so your plugin can simply instruct it to download the
dictionaries automatically on the very first run.

#+begin_src diff
--- lib/minimal_mistakes_plus/furigana.rb
+++ lib/minimal_mistakes_plus/furigana.rb
@@ -20,6 +20,12 @@
           @disabled = true
           return
         end
+
+        # Auto-install Node dependencies if missing (e.g., in GitHub Actions or fresh gem install)
+        unless Dir.exist?(File.join(File.dirname(__FILE__), 'node_modules'))
+          Jekyll.logger.info "Furigana:", "Installing Node.js dependencies. This may take a moment..."
+          system("npm install", chdir: File.dirname(__FILE__))
+        end
 
         @stdin, @stdout, @stderr, @wait_thr = Open3.popen3("node #{script_path}")
#+end_src

*** Characters not found in the dictionary

Some characters are parsed incorrectly, such as 箇 and 个:

#+begin_src html
<p>Some characters are parsed incorrectly, such as <ruby furiganated="true">箇<rp>(</rp><rt class="frgntr_unselectable">箇</rt><rp>)</rp></ruby> or <ruby furiganated="true">个<rp>(</rp><rt class="frgntr_unselectable">个</rt><rp>)</rp></ruby>:</p>
#+end_src

whose Furigana is added as itself. Is this caused by the kuroshiro backend or
our own codes?


This happens because of the underlying Kuromoji dictionary used by the Kuroshiro
backend.

When Kuromoji encounters a single kanji character or variant that it doesn't
have a specific contextual reading for (or treats as an unknown word), it
defaults to returning the surface character itself as the reading. Kuroshiro
then sees a kanji character, sees a reading attached to it, and faithfully wraps
it in a ~<ruby>~ tag, resulting in ~<ruby>箇...<rt>箇</rt>...</ruby>~.

We can easily fix this by filtering the HTML output. We'll add a cleanup step to
strip the <ruby> tags whenever the base text perfectly matches the reading text.
We need to do this on both the Ruby backend (for the static build) and the
JavaScript frontend (for the live converter).

** Bug fixes
*** Kuromoji Resources Not Found

{: .notice}
On the GitHub Actions webpage when clicking on the Furigana Converter input box,
the whole page freezes. Errors reporting kuromoji resources 404 (Not Found) can
be seen in the console, as shown in the screenshot. It works fine when running
locally. I think it's because the CDN links are wrong, such as
https://josephtesfaye.com/cdn.jsdelivr.net/npm/kuromoji@0.1.2/dict/base.dat.gz.
How does this happen? Fix it.

Gemini (Pro):

This is a deep, notorious quirk caused by a combination of Kuromoji's internal
code and the browser's URL resolution rules. Here is exactly what is happening:

1. The ~path.join~ Bug: Inside the Kuromoji library, the loader combines your
   dictPath with the filename using a browser polyfill of Node's ~path.join()~.
   This polyfill strictly normalizes paths and strips out double slashes. It
   takes "https://cdn.jsdelivr.net/npm/kuromoji@0.1.2/dict/" and turns it into
   "https:/cdn.jsdelivr.net/npm/kuromoji@0.1.2/dict/base.dat.gz" (with only one
   slash after =https:=).

2. Why it worked locally (HTTP): Your local server runs on =http://=. When the
   browser's XMLHttpRequest tries to fetch =https:/cdn...=, it sees that the
   scheme (=https=) doesn't match your local server (=http=). Because it's a
   cross-scheme request, the browser safely auto-corrects the missing slash and
   fetches the dictionary from the external CDN.

3. Why it failed in Actions (HTTPS): GitHub Pages runs on =https://=. When the
   browser sees the request to =https:/cdn...=, the schemes match perfectly.
   According to standard web specification, a matching scheme with a single slash
   is treated as a relative absolute-path on the current domain. Therefore, the
   browser forcefully appends it to your site, resulting in the 404 error at
   https://josephtesfaye.com/cdn.jsdelivr.net/...!

* Translator
** Requirements

Now I also want Furigana Converter to be able to show translation (in English or
a selected language) for the input text. The source text (with furigana) and
translation are shown in pairs sentence by sentence, with the translation on a
new line just below the source.

Similarly, these options are used in the front matter:

#+begin_src org
translate: true/false
translate_targets: [English, Chinese]
#+end_src

- ~translate~: Toggle the translation on (true) or off (false or unset).
- ~translate_targets~: Specify the languages into which the converter can translate.

When translation is toggled on, show an icon in a TRANSLATION row just above the
FURIGANA row in the toolbar. When the icon is clicked a dropdown list of the
~translate_targets~ values is shown in the same order, and the icon changes to
correspond to the selected language (i.e., each language has its own icon). You
can select none of the languages, in which case the translation is toggled off
and the icon changes to a corresponding look. The first value in
~translate_targets~ is the selected language by default, not OFF.

But before we start coding I want to discuss the feasibility of this feature. At
first thought, I want the translation to work offline. The question is, is there
a mature offline translator which gives you reliable translations? Would it be
too large and too reliant on computational power to be loaded via CDN and hence
impractical for my purpose? If we have to use online translators such as Google,
Microsoft, DeepL, etc, do they provide APIs for third-party usage? Finally, can
we use AI services from Google, OpenAI, etc.? But I want to keep the translation
feature lightweight and free (no cost).

** Discussion 1

This is an excellent architectural question. Adding live translation to a purely
static site (like Jekyll on GitHub Pages) introduces a very specific set of
constraints because you do not have a backend server to process data or hide
secrets.

Here is a breakdown of the feasibility of each approach, balancing your
requirements for reliability, cost, and payload size.

*** The Offline Approach (Client-Side AI)

It is actually possible to run translations entirely offline in the browser
using WebAssembly and libraries like Hugging Face's =transformers.js=. * *How it
works:* The browser downloads a quantized translation model (like
Opus-MT/MarianMT or a tiny LLM) via CDN and runs the neural network directly on
the user's CPU/GPU. * *The Problem (Size):* These models are inherently heavy. A
specialized translation model (e.g., Japanese to English) is typically *70MB to
150MB*. If you want Japanese to Chinese as well, that is another 70MB+ download.
​* *Verdict:* While it is completely free and private, it drastically violates
the "lightweight" requirement. It will freeze the browser tab during
initialization and consume significant mobile data.

*** Traditional Cloud APIs (DeepL, Google Translate, Microsoft)

These services provide the most reliable, fastest, and accurate translations.
They all offer APIs and generous free tiers (e.g., DeepL offers 500,000
characters per month for free). * *The Problem (Security):* To use these APIs,
you must send an API key with your request. Because your site is static HTML/JS,
you would have to hardcode your API key into your JavaScript file. * *The Risk:*
Bots constantly scrape GitHub and public websites for exposed API keys. Within
hours, someone could steal your key, blow through your free tier limit, and
either lock your account or charge your credit card. * *Verdict:* Never put API
keys in client-side JavaScript. This makes direct API integration impossible for
a static site without a proxy.

*** AI Services (OpenAI, Google Gemini, Anthropic)

Using LLMs provides fantastic, context-aware translations (often better than
traditional translators for Japanese). Google Gemini, for instance, has a very
generous free tier. * *The Problem (Security):* Exactly the same as the
traditional APIs. Exposing an OpenAI or Gemini API key in your client-side
JavaScript is a massive security risk. Both Google and OpenAI actively scan
GitHub and will automatically revoke your key the moment they detect it in your
public repository. * *Verdict:* Unusable directly from the browser for public
static sites.

*** Free, Keyless Web APIs (The "Hack" Approach)

There are community-hosted proxy APIs (like Lingva Translate or public instances
of LibreTranslate) that attempt to scrape Google Translate or run open-source
models for free without requiring API keys. * *The Problem (Reliability &
CORS):* These free instances are constantly overloaded, frequently go offline,
and often suffer from CORS (Cross-Origin Resource Sharing) blocks, meaning your
browser will refuse to load the data for security reasons. Furthermore, if you
try to hit Google Translate's web endpoints directly via JS, Google will block
you with a =429 Too Many Requests= or CORS error. * *Verdict:* Too unreliable
for a polished feature.

*** Recommended Solutions

Since a purely static frontend cannot safely hold API keys, and local browser
models are too large, you have two realistic paths forward:

*Path A: The Serverless Proxy (Most Reliable & Free)* You can easily spin up a
free "Serverless Function" using Cloudflare Workers, Vercel, or Netlify. 1. You
create an account on Cloudflare (free) or Vercel (free). 2. You write a 20-line
JavaScript function that holds your Google Gemini or DeepL API key securely on
/their/ servers. 3. Your Furigana Converter sends the Japanese text to your
serverless function URL. 4. The function talks to Gemini/DeepL, gets the
translation, and sends it back to your site. * *Pros:* 100% secure, highly
reliable, and remains entirely free (using Cloudflare's free tier +
Gemini/DeepL's free tier). Keeps your frontend incredibly lightweight. * *Cons:*
Requires setting up an external account (Cloudflare/Vercel) outside of your
Jekyll repository.

*Path B: The Browser Native API (The Future, but Experimental)* Google Chrome
has recently started building an AI translation model directly into the browser
itself (_{window.translation} API). * *Pros:* Zero network requests, instant,
completely free, zero download (it's built into Chrome). * *Cons:* It is
currently an experimental feature. It only works on recent versions of
Chrome/Edge, requires users to have certain flags enabled, and does not work on
Safari or Firefox yet.

If you want a robust, production-ready feature today that stays free and
lightweight, *Path A (Serverless Proxy with a free Gemini or DeepL API key)* is
the standard architectural best practice. We can easily write the proxy code and
the frontend Jekyll integration if you are open to using Cloudflare Workers or
Vercel.

** Discussion 2

Is there non-LLM or non-AI solution for local translation, like a command-line
tool or something whose output can be utilized by other apps? What are the best
ones?

To answer this, we need to clarify a quick technical distinction: truly "non-AI"
translation is called Rule-Based Machine Translation (RBMT). RBMT relies
entirely on hardcoded grammatical dictionaries. For structurally identical
languages (like Spanish to Catalan), it works okay. For Japanese to English,
RBMT produces completely unintelligible gibberish.

However, if by "non-LLM" you mean you want to avoid massive, resource-heavy
models (like ChatGPT or LLaMA) and instead want a *lightweight, dedicated
translation engine* that runs locally on your CPU via the command line, there
are excellent options.

Here are the best command-line translation tools whose outputs can be easily
piped into other applications or parsed by a Ruby script.

*** Argos Translate (True Local Offline)

Argos Translate is an open-source offline translation engine built on OpenNMT
(Neural Machine Translation). It is exactly what you are looking for: it uses
small, highly optimized models specifically trained for single language pairs
(about 30MB to 50MB per language), requiring no GPU and running entirely
offline.

- *How it works:* It is a Python package that installs a command-line utility.
- *CLI Usage:* =argos-translate --from ja --to en "こんにちは"=
- *Integration:* You can easily call it from your Jekyll Ruby plugins using
  =Open3.capture2=, capturing the standard output directly into your HTML
  generation process.

*** Translate-Shell (The API Wrapper)

If you want the high accuracy of Google Translate, DeepL, or Bing without
dealing with API keys or cloud accounts, =translate-shell= (formerly Google
Translate CLI) is the gold standard for command-line users.

- *How it works:* It is a pure shell script (Awk) that acts as a command-line
  interface for public web translators. It scrapes the web endpoints invisibly.
- *CLI Usage:* =trans -b ja:en "本日"= (The =-b= flag returns "brief" output,
  which is just the raw translated string, perfect for scripts).
- *Integration:* Extremely fast and native to Unix environments. The only
  downside is that if your build script sends thousands of requests in a few
  seconds, the IP might temporarily be rate-limited by the engines.

*** Bergamot / Marian NMT (The Browser-Native Engine)

Marian NMT is the engine written in pure C++ that powers Mozilla Firefox's new
local, offline translation feature. It is incredibly fast and designed
specifically for consumer CPUs rather than server GPUs.

- *How it works:* While it has a command-line interface, its real power is that
  it is compiled to WebAssembly.
- *Integration:* This is actually what you would use if you wanted to embed an
  offline translator directly into a webpage without a server. However, as
  discussed previously, the WASM binaries and language packs still require the
  user to download around 30MB-50MB of files when they visit the page.

*** The Architectural Reality Check for Jekyll

While these command-line tools are perfect for local text processing, we need to
address how they fit into your specific Jekyll environment:

*Scenario A: Build-Time Translation (Works perfectly)* If you are translating
the static content of your posts (e.g., adding =translate: true= to the front
matter and having a Ruby plugin parse the Japanese text and generate English
text below it), these CLI tools are the perfect solution. Your =furigana.rb=
script can pass the Japanese strings to =argos-translate= via the shell, and
bake the English text permanently into the final HTML during the build phase.

*Scenario B: Live Converter Input Box (Will not work)* If your goal is for the
*Live Furigana Converter* to translate text that a user pastes into the browser
/after/ the site has been published, a local CLI tool cannot help you. 1. GitHub
Pages only hosts static HTML/JS; it cannot run your local Mac command-line
tools. 2. The user's browser cannot execute shell scripts.

For a live, in-browser input box to generate translations instantly on a static
site, you are strictly limited to either injecting a heavy WebAssembly model
into the client's browser, or passing the text to an external cloud proxy via
JavaScript.

*** Why =translate-shell= Doesn't Need a Key

=translate-shell= bypasses the need for an API key by acting as a web scraper.
Instead of communicating with Google's official developer API, it constructs an
HTTP request that mimics a normal user typing text into the public
=translate.google.com= website. It then intercepts and parses the raw JSON
responses that Google's frontend sends back to the browser.

Because it pretends to be regular web traffic, Google provides the translation
for free, just as it would if you manually visited the site.

*** Why We Can't Replicate This in the Browser

If =translate-shell= can scrape the public website for free, you might
logically wonder why our client-side JavaScript can't just do the exact same
thing.

The barrier is *CORS* (Cross-Origin Resource Sharing). Modern browsers enforce
strict security policies to prevent malicious scripts from interacting with
other websites behind the scenes. If your JavaScript on =josephtesfaye.com=
tries to silently fetch data from =translate.google.com=, the browser will
instantly block the request because the origins do not match. =translate-shell=
runs in your terminal, which does not enforce browser security policies like
CORS, giving it total freedom to scrape data.

*** The Google Translate Widget

The widget you are remembering is the *Google Translate Website Translator*.
While it seems like a perfect fit, there are two major issues with using it for
your specific use case:

1. *It is Deprecated:* Google officially retired this tool for new websites
   in 2019. While legacy scripts still float around the internet and technically
   function, it is unsupported, unmaintained, and could be completely shut down
   by Google at any time.
2. *Architectural Incompatibility:* The widget is designed to forcefully mutate
   the DOM and replace the text /in place/. It does not provide an API to simply
   "return" the translated string back to your code. Because your goal is to
   pair the source text and translation sentence by sentence (placing the
   English on a new line below the Japanese), the widget will fail. It would
   simply overwrite your Japanese text with English text rather than giving you
   the raw string to format yourself.

/(Note: You can technically restrict the widget to a certain area by wrapping
the rest of the page in a _{notranslate} CSS class, but it still suffers from
the DOM-mutating behavior mentioned above.)/

*** The Best Approach Forward

To achieve your exact layout---a clean, instant translation returned as a string
that we can format underneath the Furigana---without exposing API keys or
battling CORS blocks, we need a proxy.

The fastest, most efficient, and best approach is to create a *Cloudflare
Worker*. It is a 100% free, serverless function that takes minutes to set up. 1.
Your frontend sends the Japanese text to your Worker URL. 2. The Worker (which
safely holds a free Gemini or DeepL API key on Cloudflare's secure servers)
requests the translation. 3. The Worker returns the pure translated string back
to your Jekyll site for formatting.
